1 · Reading & focus order

docsopt-in later

The most important one to understand. The grid looks correct, but a keyboard or screen-reader user travels through it in a different order than your eyes do.

What your eyes see (visual order)

Reads left→right, top→bottom: 1, 2, 3, 4, 5 …

What Tab / a screen reader follows (DOM order)

Runs down each column: 1, 4, 7 → 2, 5, 8 → …

Watch the highlight jump down column 1, then column 2…
Why it happens: to guarantee a tile can never be clipped, each column is its own DOM container and items are placed into those columns. So the DOM ends up column-major even though the visuals are row-major. The fix isn't free — it's the trade-off of this whole approach. For a photo wall it's harmless; for cards with headings/prices/links it can violate WCAG 1.3.2 (Meaningful Sequence). Plan: (1) document it loudly now, (2) offer an opt-in single-flow mode, (3) adopt native grid-lanes which preserves source order (see #11).

2 · Pass-through props (class, style, id, …)

core

Right now the component renders a fixed <div class="masonry"> and gives you no way to hook your own styling, ids, or attributes onto it. "Pass-through" just means: forward any extra attributes you put on the component straight onto that root element.

Today — can't reach the root

<!-- class is ignored; nowhere to put an id -->
<Masonry class="gallery" id="feed" … />

You end up wrapping it in an extra <div> just to style it.

With pass-through

<Masonry
  class="gallery"
  id="feed"
  data-testid="grid"
  …
/>
// → all land on the real root element

Style and target it directly. No wrapper needed.

Tiny change, big ergonomics win — this is the #1 thing that makes a library feel "usable in any repo" vs. "I had to fork it."

3 · heightEstimate() — one escape hatch

opt-in

Reminder: the library guesses each tile's height only to decide which column it joins. Today that guess is built from two separate props. This feature replaces them with one optional function for full control — while keeping the simple props as sugar.

How the height guess is assembled today

The limitation

footerEstimate is a single flat number for every tile. But a card with a 1-line caption and one with a 4-line caption have very different chrome. You can't express that today.

One general function

// total estimate for this item, your call:
heightEstimate={(item, colWidth) =>
  colWidth / item.ratio        // image
  + item.lines * 20          // caption
  + (item.hasChips ? 32 : 0) // chips
}
aspectRatio + footerEstimate stay as the easy path; heightEstimate is the "I need precise balancing" override. One concept, fully general, still measurement-free.

4 · Separate columnGap / rowGap

opt-in

Today one gap controls both the space between columns and between stacked tiles. Some designs want them different (tight columns, airy rows). This splits them, keeping gap as a shorthand.

One gap (today)

gap = 14 everywhere

Independent gaps

columnGap = 24, rowGap = 6

5 · onlayout / oncolumnchange callback

opt-in

A small notification: the component tells you whenever it recalculates — how many columns it chose and how wide they are. It changes no behavior; it just lets your code react (analytics, "showing N columns" labels, triggering image preloads, etc.).

6 · Kill the SSR flash (initialColumns)

opt-in

On the server there's no window to measure, so width starts at 0 and the grid renders as a single tall column. The browser then measures and reflows into the real columns — a visible "flash" / layout shift on first paint. This feature lets you tell it how many columns to assume up front.

First-paint timeline

Alternative for the no-JS frame: render a CSS columns fallback until measurement kicks in. Either way the goal is the same — no 1-column flash.

7 · Explicit breakpoint columns

opt-in

Today column count is derived from minColumnWidth — "fit as many ~200px columns as you can." That's fluid and great by default. But some designs need exact counts at specific widths. This adds an alternative: a width→count map.

Derived (today)

<Masonry minColumnWidth={200} … />

"As many 200px-ish columns as fit." Count changes smoothly with width.

Explicit breakpoints

<Masonry columns={{ 0: 1, 600: 2, 1000: 4 }} … />

"1 column under 600px, 2 up to 1000px, 4 beyond." Pixel-exact, designer-friendly.

8 · Empty & loading slots

opt-in

A convenience: built-in places to render "nothing here yet" or a skeleton, instead of every caller writing their own {#if items.length === 0} wrapper. 

<Masonry items={photos} …>
  {#snippet empty()}   // shown when items is empty
    <p>No photos yet.</p>
  {/snippet}
  {#snippet children(photo)} … {/snippet}
</Masonry>

9 · Virtualization (windowing)

add-on

The big-scale feature. Today every item becomes a DOM node — fine for a few hundred, painful for thousands. Virtualization renders only the tiles currently near the viewport and recycles them as you scroll, so 10,000 photos cost the same as ~30.

Today — render everything

Virtualized — render the window

Why it's an add-on, not core: to skip rendering off-screen tiles, you must reserve their space — which means knowing/measuring their heights and absolutely positioning them. That's exactly the "fixed-frame" technique this library deliberately avoids (it's what can clip a tile). So virtualization lives in a separate opt-in module (e.g. @nicbat/svelte-masonry/virtual) with its own trade-offs, leaving the core clip-proof.

10 · FLIP reflow animation

opt-in

When the column count changes (e.g. you resize past a breakpoint), every tile instantly jumps to its new spot. FLIP is a technique that smoothly animates those jumps instead. Press play to compare.

Without (today) — instant jump

With FLIP — animated

Built into the core as a boolean (animate={true}, default false). FLIP measures positions and animates with transform — a compositor-only effect that never participates in layout, so it can't clip a tile and is fully compatible with the invariant (unlike virtualization). Off-cost: zero behavior + ~1 KB. The one wrinkle: tiles live in separate per-column {#each} blocks, so cross-column moves need a small manual FLIP rather than Svelte's built-in animate:flip directive.

11 · Native grid-lanes backend

add-on

Browsers are adding a built-in masonry mode (display: grid-lanes). When it's widely supported, the same component could emit native lanes instead of JS-managed columns — same props, but the browser does the packing, and (bonus) source order is preserved, fixing the reading-order issue from #1.

Not yet: as of 2026 grid-lanes ships only behind browser flags. This is a "design the seam now, flip it on later" item — and native still won't give you content-balancing or virtualization, so the library keeps earning its place.

12 · Deliberate non-goals

non-goal

Staying minimal means saying no on purpose. These belong to heavier libraries (e.g. Muuri) and would betray the dependency-free, clip-proof promise:

✗ Drag-and-drop reordering

Needs absolute positioning + hit-testing; a different kind of component.

✗ Built-in filter / sort animations

Show/hide choreography is app-specific and heavy; compose it outside.

Documenting non-goals is a feature too — it tells contributors (and future you) where the line is, so the core stays tiny.
